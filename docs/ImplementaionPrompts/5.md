Build a .NET 8 Console App that acts as an MCP client runner/orchestrator for our local MCP servers.

The app must:
- Start each MCP server as a child process (dotnet run or executable command).
- Communicate with each server via stdin/stdout using JSON-RPC 2.0.
- Route tools/call requests to the correct server based on tool name.
- Support tool discovery (tools/list) and caching.

Scope (ONLY these services):
1) OnlineTeamTools.MCP.KieAi
2) OnlineTeamTools.MCP.Media
3) OnlineTeamTools.MCP.YouTube
Exclude OnlineTeamTools.MCP.Jira entirely (no references, no config, no routing).

========================
Transport / Protocol
========================
- JSON-RPC 2.0 over stdio.
- Supported methods:
  - tools/list
  - tools/call
- Request formats (exact):
  {"jsonrpc":"2.0","id":<string|number>,"method":"tools/list","params":{}}
  {"jsonrpc":"2.0","id":<string|number>,"method":"tools/call","params":{"name":"<tool_name>","arguments":{...}}}

Line protocol:
- Write one JSON request per line to stdin (newline-delimited JSON).
- Read one JSON response per line from stdout (newline-delimited JSON).
- stderr is NOT JSON; capture and log it separately for diagnostics.
- Do not mix logs with stdout JSON in the runner itself:
  - stdout: only final command output JSON (for CLI user)
  - stderr: logs/diagnostics

Correlation / reliability:
- Correlate responses by id (concurrent calls allowed).
- Per-call timeout and CancellationToken support.
- If the child process exits unexpectedly:
  - restart it automatically (with backoff)
  - fail any in-flight requests with a clear error
  - allow subsequent calls to succeed after restart

========================
Normalization Rule (Important)
========================
Result payload shape differs by service:
- KieAi + Media: tool payload is returned in `result` directly.
- YouTube: tool payload is returned in `result.data`.
Normalize in the runner so the CLI always prints:
  { "ok": true/false, "data": <tool_payload>, "meta": {...optional...} }
Where:
- data = tool payload from KieAi/Media `result`, OR from YouTube `result.data`
- preserve raw response optionally behind a flag (e.g. --raw)

========================
Service Registry & Routing
========================
Create a Service Registry that defines:
- Service name
- Process start command (FileName + Arguments, working directory if needed)
- Environment variables
- Optional tool allow-list (or dynamic discovery + cache)

Routing rules:
- Prefer dynamic discovery:
  - On startup (or first request), call tools/list for each service and cache tool names -> service mapping.
  - Provide a manual override mapping in config (optional).
- When a tools/call arrives:
  - look up tool_name -> service
  - forward request to that service
  - normalize response shape (see above)

========================
CLI Commands
========================
Provide a CLI with commands:

1) list-tools <service|all>
- Calls tools/list on one service or all services.
- Prints normalized JSON.

2) call <tool_name> --args '<json>'
- Routes by tool_name.
- --args is a JSON object string (must parse).
- Prints normalized JSON result.
- Support --timeout <seconds>.
- Support --raw to print the raw JSON-RPC response.

Exit codes:
- 0 on success (tool call ok == true)
- non-zero on failure (transport errors, timeouts, tool ok == false)

========================
Architecture / Code Requirements
========================
Keep the code structured and testable:
- ProcessHost (start/stop/restart, stderr capture)
- JsonRpcClient (send/receive NDJSON, correlation, timeouts)
- ToolRegistry/Discovery (tools/list cache)
- ToolRouter (tool_name -> service)
- CLI command handler

Non-goals:
- No Jira integration.
- No HTTP transport.
- No UI.

Acceptance criteria:
- Can start all 3 child MCP servers and list tools.
- Can route a tools/call by name to the correct service.
- Handles concurrent calls without id collisions.
- Clean JSON output to stdout; diagnostics to stderr.
- Restarts a crashed child process and continues working.
